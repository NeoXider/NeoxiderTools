## Карта урока

| Что | Ответ |
|-----|-------|
| Объект работы | Network traffic, spawn counts, send rate, allocations. |
| Кто владеет state | Server decides what to send; profiler decides what to optimize. |
| Как проверить | Baseline до/после на одинаковом сценарии. |
| Артефакт | Таблица profiling: metric, before, after, decision. |

---

## Что должно получиться

Вы умеете доказать, что оптимизация помогла: есть замер до, изменение, замер после и вывод.

---

## Проблема

"Оптимизация на глаз" часто делает код сложнее и не уменьшает трафик. В сетевой игре сначала надо найти, что именно дорого.

---

## Что измерять

| Метрика | Зачем |
|---------|-------|
| bytes/sec per client | Стоимость канала. |
| messages/sec | Спам и частота. |
| spawned objects | Масштаб observers. |
| GC alloc | Статтеры на клиенте/сервере. |
| CPU server frame | Сколько матчей выдержит сервер. |

---

## Практика

1. Выберите один повторяемый сценарий: 2 клиента, 10 NPC, 60 секунд.
2. Запишите baseline: bytes/sec, messages/sec, GC.
3. Найдите самый дорогой тип state.
4. Примените одну оптимизацию.
5. Повторите тот же сценарий.
6. Запишите вывод.

Пример оптимизаций:

| Симптом | Возможное решение |
|---------|-------------------|
| Слишком много transform updates | Больше sync interval, отключить лишние оси. |
| Всем уходят далёкие объекты | Interest Management. |
| Частые float-изменения | Квантизация/порог изменения. |
| Инвентарь пересылается целиком | Sync-коллекции. |
| UI пересобирается полностью | Diff update. |

---

## Проверка себя

В отчёте есть:

- точный сценарий;
- версия Mirror;
- количество клиентов;
- цифры до;
- изменение;
- цифры после;
- вывод, стоит ли оставлять оптимизацию.

---

## Минимальная диагностика

| Симптом | Что проверить |
|---------|---------------|
| Трафик растёт с каждым игроком слишком быстро | Interest Management, observers, broadcast RPC. |
| GC spikes | Allocations в update/serialization/UI redraw. |
| Spawn storm | Pooling, spawn frequency, object lifetime. |
| Непонятно, что оптимизировать | Нет baseline scenario и метрик. |

---

## Частые ошибки

- Менять сразу пять параметров и не понимать, что помогло.
- Оптимизировать до первого playable build.
- Уменьшать трафик ценой поломанной честности.
- Игнорировать GC на клиенте.

---

## Лайфхаки

- Самая сильная оптимизация часто "не отправлять тем, кому не нужно".
- Не синхронизируйте scale/rotation, если они не меняются.
- Для UI используйте object pooling и diff.
- Храните отчёты в `NETWORK_PROFILING.md`.

---

## Профессиональный минимум

- У каждой оптимизации есть измеримый before/after.
- Send Rate и sync fields настроены под gameplay, а не оставлены случайно.
- Broadcast заменён на targeted/interest-based flow, где это важно.
- Профилирование повторяется после Mirror upgrade.
