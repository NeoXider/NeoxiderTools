## Цели

Сделать сервер **наблюдаемым**: что писать в лог, какие счётчики держать, когда выносить прогресс в БД; связать эксплуатацию с Mirror (`connectionId`, жизненный цикл матча) без утечки персональных данных в plain text.

---

## Проблема и контекст

| Только `Debug.Log` в файле | Нет ротации, диск забивается, нет корреляции по матчу |
|----------------------------|--------------------------------------------------------|
| Лог без `matchId` | Невозможно восстановить хронологию инцидента |
| Метрики «на каждый кадр» | Сам сервер деградирует от наблюдателя |

---

## Теория

**Структурированные логи** — это не «красивый JSON ради моды», а возможность за минуту ответить: «что случилось с матчем M между T0 и T1 для соединения C?». Минимальный набор полей: время UTC, build id, match id, `connectionId`, уровень, короткий код события, **без** огромных payloadов `SyncVar`.

**Метрики**: CCU, длительность тика симуляции, время обработки сетевого цикла (осторожно: замер сам по себе стоит CPU). Для ранней стадии достаточно **счётчиков в памяти** + редкий дамп в файл; Prometheus — когда есть кто поддерживать дашборды.

**Persistence**: долгоживущий прогресс (инвентарь, платежи) — **вне** Mirror как источника истины. Mirror держит **сессию**; SQL/HTTP/облако — **аккаунт**. См. урок 19 про токены и сессии.

---

## Практика

1. Обёртка `GameLogger.Info/Warn` с **throttle** на hot path (не логировать каждый фикс физики).
2. Эндпоинт health: «процесс жив, слушает порт» — для балансировщика и оркестратора (урок 25).
3. Определите **retention**: сколько дней хранить файлы на диске VPS до ротации.

### Шаг 1 — контракт строки лога

Зафиксируйте в `OPS.md` таблицу полей. Пример (идея, не стандарт индустрии):

| Поле | Зачем |
|------|--------|
| `ts` | Время UTC ISO8601 |
| `lvl` | INFO/WARN/ERROR |
| `build` | Git short hash |
| `match` | UUID матча |
| `conn` | `connectionId` Mirror |
| `evt` | Короткий код: `JOIN`, `DROP`, `SNAP_ERR` |
| `msg` | Человекочитаемое одно предложение |

### Шаг 2 — корреляция с Mirror

На `OnServerConnect` / `OnServerDisconnect` логируйте **одну** строку с `connectionId` и `address` (осторожно с GDPR — иногда достаточно хэша). На спавне игрока — связь `conn` → `playerEntityId` вашей доменной модели.

### Шаг 3 — health HTTP

Минимальный `HttpListener` или встроенный в ваш backend микросервис: `GET /health` → 200 если `NetworkServer.active` и порт открыт. Не путайте с игровым UDP/TCP портом Mirror — health может быть отдельным TCP.

---

## Типичные ошибки

- Логировать весь `SyncVar` payload каждый кадр — диск и I/O убивают матч.
- Писать **секреты** токенов в лог даже при `Debug` — логи утекают в бэкапы.
- Один гигантский файл без ротации — ночью диск full → сервер падает.

---

## Советы и паттерны

Отправка метрик в Prometheus / облако — по мере роста; начните с файл + простого HTTP ping.

Используйте **уровни**: `ERROR` только для действий, требующих pager duty; иначе усталость от алертов.
