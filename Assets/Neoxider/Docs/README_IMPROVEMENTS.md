# Рекомендации по улучшению README и документации

Краткий чеклист и идеи для главного README и раздела Docs.

---

## 1. Главный README (`Assets/Neoxider/README.md`)

### Структура и навигация
- **Оглавление (TOC)** в начале — при длине >100 строк добавьте якорные ссылки (`## Установка` → `#установка`), чтобы быстро переходить к установке, модулям, примерам.
- **Один «вход» в доки** — в README оставить блок «Документация» с одной ссылкой на `Docs/README.md` (индекс модулей), а полную таблицу модулей хранить только в `Docs/README.md`, чтобы не дублировать и не расходиться.

### Ясность для новых пользователей
- **Первый запуск (чеклист)** — явный список: импорт пакета → добавление `--System--.prefab` → один пример (например, Counter или NeoCondition). Можно вынести в отдельный короткий файл `Docs/QuickStart.md` и дать ссылку из README.
- **Требования в одном месте** — Unity 2022.1+, опционально DOTween/Odin — оформить отдельным подразделом «Требования» с таблицей (как сейчас зависимости), чтобы не искать по тексту.

### Язык и аудитория
- **Единый язык** — сейчас README на русском, часть тултипов переведена на английский. Имеет смысл выбрать политику: либо README + Docs полностью на русском, либо добавить английскую версию `README.en.md` (хотя бы установка + модули + ссылка на Docs) для поиска и GitHub.

### Визуал и бейджи
- **Бейджи** (опционально) — версия пакета, минимальная версия Unity, лицензия — в одну строку под заголовком, чтобы сразу было видно совместимость.
- **Скриншот/гиф** — один пример (например, NeoCondition в Inspector или демо-сцена) повышает понимание «что это за пакет».

### Примеры и код
- **Единый стиль примеров** — в README уже есть пример Save (C#) и пошаговые примеры (Counter, Timer). Можно добавить один минимальный пример для NeoCondition в виде кода (как подписаться на событие из скрипта), если целевая аудитория — программисты.

## 2. Документация (`Docs/`)

### Единый индекс
- **Один индекс модулей** — в `Docs/README.md` уже есть полная таблица. Главный README может содержать сокращённую таблицу (топ-5–7 модулей) + ссылку «Все модули → Docs/README.md», чтобы не дублировать и не забывать обновлять два места.

### Шаблон страницы модуля
- **Единый шаблон** для каждого компонента/модуля:
  1. **Назначение** — одна фраза, для чего компонент.
  2. **Установка/подключение** — если нужно (asmdef, префаб).
  3. **Основные поля (Inspector)** — таблица или список с краткими описаниями.
  4. **API (опционально)** — ключевые методы/события; детали можно оставить в XML в коде.
  5. **Пример использования** — сцена, префаб или фрагмент кода.
  6. **См. также** — ссылки на связанные документы (например, TimerObject → CooldownReward, Save).

Так проще поддерживать консистентность и не пропускать разделы.

### Навигация внутри Docs
- **Обратные ссылки** — в конце глубоких статей (например, `TimerObject.md`, `CooldownRewardExtensions.md`) добавлять «← [Tools/Time](README.md)» или «← [Bonus](README.md)», чтобы было проще вернуться к разделу.
- **Версия/дата** — в критичных файлах (например, миграция TimeReward → CooldownReward) указать версию пакета (v6.0.0) или дату последнего обновления, чтобы было понятно, к какой версии относится описание.

### Связность и дублирование
- **Перекрёстные ссылки** — где модули зависят друг от друга (Save, TimerObject, CooldownReward, NeoCondition), в тексте давать явные ссылки на соответствующие .md.
- **Устаревшие компоненты** — как с TimeReward: в начале документа помечать **[Obsolete]** и давать ссылку на замену (CooldownReward) и краткую инструкцию по миграции.

### Поиск и целостность
- **Проверка ссылок** — периодически проверять, что все внутренние ссылки (на другие .md) ведут на существующие файлы; при необходимости — скрипт или ручной чеклист перед релизом.
- **Поиск по документации** — на GitHub можно искать по репозиторию; если позже будет отдельный сайт документации, имеет смысл полнотекстовый поиск по разделам.

### Сложные модули
- **Схемы и диаграммы** — для StateMachine, Condition, цепочки Save → TimerObject → CooldownReward можно добавить простые Mermaid-диаграммы (поток данных, состояния), они хорошо рендерятся на GitHub.

## 3. Приоритеты (что сделать в первую очередь)

| Приоритет | Действие |
|-----------|----------|
| Высокий   | Добавить TOC в главный README; сократить таблицу модулей в README и вести полный список только в Docs/README.md. |
| Высокий   | Оформить единый раздел «Требования» в README и убедиться, что ссылки на CHANGELOG и PROJECT_SUMMARY актуальны. |
| Средний   | Ввести минимальный шаблон для новых документов (Назначение → Поля → API → Пример → См. также). |
| Средний   | В ключевых doc-файлах добавить «См. также» и обратные ссылки на родительский раздел. |
| Низкий   | Английская версия README (хотя бы установка + модули); бейджи; один скриншот/гиф. |
| Низкий   | Mermaid-диаграммы для StateMachine/Condition; проверка битых ссылок скриптом. |
