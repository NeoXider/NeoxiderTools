## Что должно получиться

Вы понимаете, как ограничить набор объектов, которые конкретный клиент получает от сервера. Это уменьшает трафик и нагрузку клиента.

---

## Проблема

Без Interest Management клиент может получать state объектов, которые далеко, в другой комнате, другой сцене или вообще не нужны игроку. На маленькой сцене это незаметно, на живом сервере становится дорогим.

---

## Теория коротко

У каждого `NetworkIdentity` есть observers: соединения, которым сервер отправляет состояние объекта. Interest Management решает, кто должен быть observer.

Важно: visibility в Mirror не равно "видно камерой". Это сетевой интерес. Объект может быть за стеной, но всё ещё быть интересен.

Документация Mirror перечисляет built-in варианты: Distance, Spatial/Hex Spatial Hashing, Scene, Scene Distance, Match, Team и Custom Interest Management. Не выбирайте самый сложный вариант первым: начните с того, который прямо совпадает с вашей игровой структурой.

---

## Когда нужен IM

| Ситуация | Решение |
|----------|---------|
| Большая карта | Distance/Spatial interest. |
| Комнаты/матчи | Scene/Match interest. |
| Additive-сцены | Scene distance или кастомная схема. |
| Скрытая информация | Owner sync / custom visibility, плюс серверная валидация. |

Interest Management не заменяет античит. Если клиент не видит сундук, сервер всё равно проверяет дистанцию при попытке открыть.

---

## Практика

1. Создайте тестовую сцену с 20-50 сетевыми объектами.
2. Запустите Host + Client.
3. Замерьте трафик/количество spawned objects без IM.
4. Добавьте подходящий Interest Management component.
5. Настройте дистанцию или сценовую принадлежность.
6. Повторите замер.

В Mirror `96.10.0` в GitHub Releases отмечен фикс `SetHostVisibility` для Canvas renderers. Если локально стоит `96.0.1`, учитывайте разницу при объектах с world-space UI.

---

## Проверка себя

- Далёкие объекты перестают быть observers для клиента.
- При возвращении в зону объект снова появляется.
- Серверная логика объекта не останавливается из-за невидимости клиенту.
- Измерение "до/после" показывает пользу.

---

## Минимальная диагностика

| Симптом | Что проверить |
|---------|---------------|
| Объект пропал навсегда | Радиус/scene membership, rebuild interval, custom visibility. |
| Трафик не изменился | Есть ли baseline и реально ли component активен на `NetworkManager`? |
| Игрок видит чужие секреты | State всё ещё sync всем observers или нужен owner/team/match split. |
| Host отличается от Client | Проверить host visibility и world-space Canvas на локальной версии Mirror. |

---

## Частые ошибки

- Включить IM без baseline-замера.
- Слишком маленький радиус, из-за чего объекты popping прямо перед игроком.
- Считать IM защитой от плохих `Command`.
- Синхронизировать приватный state всем, надеясь скрыть UI.

---

## Лайфхаки

- Делайте сетевой радиус больше визуальной дистанции.
- Для комнат проще scene/match logic, чем чистая distance.
- Логи observers полезны при "объект пропал только у одного клиента".
- Для сложных игр заведите `IM_CONFIG.md`.

---

## Профессиональный минимум

- IM настраивается после измерения, а не "на всякий случай".
- Security не зависит только от невидимости объекта.
- Для match/room игр используется Match/Scene логика, а не один огромный distance radius.
- В документации проекта есть rebuild interval, vis range и причины выбора.
