## Проблема

Мультиплеер ломается, когда игра думает как singleplayer. В singleplayer один процесс всё решает сам. В сетевой игре процессы несколько, а правда должна быть одна.

Практичное правило:

| Вопрос | Ответ по умолчанию |
|--------|--------------------|
| Кто решает HP, урон, валюту, победу? | Сервер. |
| Что делает клиент? | Читает ввод, показывает UI, просит сервер выполнить действие. |
| Что делает host? | Он и сервер, и клиент в одном процессе. |
| Чем опасен host-only тест? | Он скрывает ошибки отдельного клиента. |

---

## Минимальная теория

**Server** ведёт состояние матча. Если два клиента спорят, сервер решает.

**Client** отправляет ввод и показывает результат. Клиентскому build нельзя доверять: его можно модифицировать, замедлить, ускорить, прочитать.

**Host** удобен для коопа и раннего теста, но для честного PvP он слабее dedicated-сервера: машина игрока одновременно исполняет сервер.

**Dedicated server** - отдельный процесс без локального игрока и UI. Он дороже в инфраструктуре, но проще для честности, логов, CI и хостинга.

**TCP/UDP:** для новичка важно не название протокола, а эффект. В сети бывают задержки, потери, повторная доставка и разный порядок прихода сообщений. Поэтому не отправляйте "всё каждый кадр".

## Практика на бумаге

Заполните таблицу для своей игры:

| Действие | Клиент отправляет | Сервер проверяет | Сервер меняет |
|----------|-------------------|------------------|---------------|
| Прыжок | input / кнопку | можно ли прыгнуть | позицию/скорость |
| Покупка | `itemId` | цену, валюту, доступность | инвентарь/валюту |
| Урон | намерение атаки | дистанцию, cooldown, цель | HP цели |
| Смена ника | новый ник | длину, фильтр, права | профиль/отображение |

Если в колонке "Клиент отправляет" стоит "новое HP" или "я победил", это почти всегда ошибка.

## Частые ошибки

- Начинать с RPC до понимания authority.
- Проверять игру только в Host.
- Отправлять по сети GameObject/ScriptableObject целиком вместо ID.
- Думать, что шифрование транспорта заменяет серверную проверку.
- Считать задержку багом, а не нормальным свойством интернета.

## Лайфхаки

- Для каждой сетевой фичи пишите одну строку: `client asks -> server validates -> server changes -> clients display`.
- Сразу заведите `NETWORKING_DECISIONS.md`; туда попадут решения из курса.
- Если действие влияет на честность или экономику, сервер должен уметь отказать.

## Минимальная диагностика

Если вы не можете решить, где менять данные, задайте вопрос: "Что будет, если клиент соврёт?" Если ложь даёт преимущество, данные меняются на сервере.

| Симптом | Что проверить |
|---------|---------------|
| В таблице много `client changes` | Вернитесь к HP/валюте/победе и перенесите их на server. |
| Действие звучит как итог | Замените на намерение: не `setDamage`, а `requestAttack`. |
| Непонятно, что логировать | Логируйте request, reject reason, `connectionId`, target id. |

## Профессиональный минимум

- У каждой сетевой фичи есть owner.
- Клиент никогда не отправляет итог честного действия.
- Сервер умеет отказать и записать причину.
- Host-only проверка не считается доказательством.
