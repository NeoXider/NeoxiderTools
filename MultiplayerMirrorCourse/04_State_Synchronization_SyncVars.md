## Цели

Понимать односторонний поток сервер → клиент, использовать `hook` для UI, избегать лишних записей и гонок при ссылках на другие сетевые объекты.

---

## Проблема и контекст

| Антипаттерн | Результат |
|-------------|-----------|
| Менять «состояние боя» с клиента напрямую | Никто не узнает / чит |
| Менять `SyncVar` каждый кадр одинаковым значением | Лишний трафик (проверяйте логику) |
| Ссылка на другой `NetworkIdentity` в момент спавна | Исторически гонки; в v96 клиент применяет spawn payload позже — всё равно проектируйте порядок инициализации |

---

## Теория

`[SyncVar]` помечает поле: сервер меняет → Mirror сериализует изменения (**dirty bits**), клиенты получают обновления. **Hooks** вызываются на клиенте при изменении — удобно для `UI Slider`.

Направление только **Server → Clients** (клиент не «пишет» SyncVar в мир).

```csharp
[SyncVar(hook = nameof(OnHealthChanged))]
int health;

void OnHealthChanged(int oldV, int newV) { hpBar.value = newV; }
```

---

## Практика

1. Добавьте `SyncVar` здоровья на `NetworkBehaviour` игрока.
2. Сервер наносит урон только на сервере.
3. В hook обновите UI только у локального представления (или через отдельный View, см. урок 12).

---

## Типичные ошибки

- Hook с тяжёлой логикой каждый апдейт — вынесите в пулинг/кэш.
- Ожидание, что `SyncVar` «синхронно» на всех кадрах — учитывайте один кадр задержки.

## Углублённо: dirty bits и частота изменений

Mirror помечает поле «грязным», когда значение **реально** меняется. Если вы присваиваете то же число, сериализатор может не слать апдейт — не полагайтесь на это как на способ «пнуть» клиентов: для форс-обновления UI используйте отдельное событие или `ClientRpc`/`Message` с явной семантикой.

### Квантование

Для `float` здоровья, которое отображается полоской, часто достаточно целого процента или фиксированного шага (0.5 / 1). Это снижает дрожание UI и число сетевых дельт (урок 13).

## Практика: разделение «модель / представление»

```csharp
// Идея: hook только красит UI, не бизнес-логика
[SyncVar(hook = nameof(OnHealthChanged))]
int health;

void OnHealthChanged(int oldV, int newV)
{
    if (hpImage == null) return;
    float t = Mathf.Clamp01(newV / (float)maxHealth);
    hpImage.color = Color.Lerp(Color.red, Color.green, t);
}
```

Тяжёлую логику (звуки, камера) запускайте отложенно или из отдельного компонента, подписанного на события.

## Таблица: SyncVar vs Rpc

| Нужно | Инструмент |
|-------|------------|
| Состояние, живущее на сервере | `SyncVar` |
| Разовый эффект | `ClientRpc` / message |
| Приватный ответ одному клиенту | `TargetRpc` |

## FAQ

**Вопрос:** Можно ли `[SyncVar]` на статическом классе?  
**Ответ:** Нет, только на полях `NetworkBehaviour` в сетевом объекте.

**Вопрос:** Нужен ли hook для каждого поля?  
**Ответ:** Только если UI или локальный кэш должны отреагировать; иначе избыточно.

**Вопрос:** Как отлаживать «не приходит»?  
**Ответ:** Лог на сервере при записи, лог в hook на клиенте, проверка `isClient`/`isServer`.
