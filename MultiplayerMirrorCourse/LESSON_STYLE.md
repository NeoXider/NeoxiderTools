# Стиль уроков

Все уроки курса должны читаться одинаково. Новичок должен понимать следующий шаг, а опытный разработчик должен быстро видеть риски.

---

## Обязательная структура

```markdown
# Урок N: тема

**Навигация:** ...

| Ключевые слова | ... |

## Карта урока
## Что должно получиться
## Проблема
## Теория коротко
## Практика
## Проверка себя
## Минимальная диагностика
## Частые ошибки
## Лайфхаки
## Профессиональный минимум
## Домашнее задание
```

Если тема не требует кода, вместо `Практика` можно использовать `Схема`, `Документ`, `Flow` или `Checklist`.

`Карта урока` должна отвечать на четыре вопроса:

- какой объект или документ меняем;
- кто владеет state;
- чем проверяем: Host + Client, Dedicated + Client, MPPM или build;
- какой лог/артефакт должен остаться.

---

## Правила содержания

- Сначала результат, потом API.
- Каждая тема должна иметь проверку.
- Для сетевого вызова всегда указывать направление: client -> server или server -> clients.
- Для важных данных указывать owner: server, client, backend, UI.
- Для новичка объяснять, где именно находится компонент: scene object, player prefab, spawned prefab, NetworkManager object.
- Старые или версионные API помечать явно.
- Не писать "просто добавьте" без указания GameObject/Prefab.
- Не смешивать UI и authority.
- Не говорить "работает", если проверено только в Host.
- Не отправлять итог от клиента: отправляйте намерение, итог считает сервер.

---

## Код

Код должен быть коротким и показывать границу ответственности.

Хорошо:

```csharp
[Command]
public void CmdBuyItem(int itemId)
{
    if (!CanBuy(itemId)) return;
    ServerBuy(itemId);
}
```

Плохо:

```csharp
public void Buy()
{
    gold -= 10;
}
```

Во втором примере непонятно, где исполняется код и кто владеет правдой.

### Правило имён

| Вид | Пример | Смысл |
|-----|--------|-------|
| `Command` | `CmdRequestAttack`, `CmdTryBuy` | Клиент просит действие. |
| `ClientRpc` | `RpcPlayHitFx`, `RpcAnnounceWinner` | Сервер сообщает событие observers. |
| `TargetRpc` | `TargetShowReject` | Сервер отвечает одному клиенту. |
| Server method | `ServerApplyDamage` | Метод меняет авторитетный state. |
| Client method | `ClientRefreshHud` | Метод только отображает. |

---

## Язык

Используйте короткие фразы:

- "Клиент просит."
- "Сервер проверяет."
- "Сервер меняет state."
- "UI показывает результат."
- "Smoke должен это поймать."

Не используйте декоративные эмодзи и маркетинговые заголовки.

---

## Проверка перед коммитом

- [ ] Урок имеет проверяемый результат.
- [ ] Есть практический шаг или документ-артефакт.
- [ ] Нет client-authoritative примеров без предупреждения.
- [ ] Кодовые блоки имеют язык.
- [ ] Ссылки на соседние уроки актуальны.
- [ ] Домашнее задание связано с реальным проектом.
- [ ] В уроке указано, как проверить отдельным Client или server build.
- [ ] Есть минимум один симптом ошибки и способ диагностики.
- [ ] Версионные нюансы Mirror/Unity вынесены явно, а не спрятаны в тексте.
