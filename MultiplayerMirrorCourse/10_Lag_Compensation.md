## Что должно получиться

Вы понимаете, почему "на моём экране попал" не всегда равно "сервер засчитал". Вы умеете выбрать минимальный уровень компенсации под жанр.

---

## Проблема

Сервер видит мир в своём времени. Клиент видит интерполированную картинку с задержкой. В action-играх это создаёт ощущение несправедливости: игрок стрелял в цель, но сервер уже считает цель в другом месте.

---

## Три уровня решения

| Уровень | Когда достаточно |
|---------|------------------|
| Простые допуски | Кооп, PvE, казуальный action. |
| Server rewind для hit detection | PvP shooter / точные попадания. |
| Client prediction + reconciliation | Движение, где задержка напрямую чувствуется. |

Не начинайте с самого сложного уровня. Сначала измерьте, что ломается при 100-150 ms.

---

## Минимальная схема выстрела

```text
client: показать локальный muzzle flash
client -> server: CmdFire(clientTime, origin, direction)
server: проверить cooldown, timestamp window, позицию, направление
server: выполнить hit detection
server -> clients: результат/эффект
```

`clientTime` нельзя принимать без ограничений. Сервер задаёт максимальное окно rewind.

---

## Практика

1. Включите симуляцию задержки 150 ms и 2% loss.
2. Проверьте движение, стрельбу, взаимодействие, UI.
3. Запишите, что стало неприятным.
4. Для каждого пункта выберите самый простой фикс.

Пример таблицы:

| Механика | Симптом при 150 ms | Решение |
|----------|--------------------|---------|
| Открытие сундука | Задержка терпима | Spinner + server result. |
| Melee hit | Иногда промах визуально странный | Увеличить серверный допуск дистанции. |
| Hitscan PvP | Частые спорные промахи | Rewind hitboxes. |
| Movement | Ввод вязкий | Prediction или client authority + проверки. |

---

## Проверка себя

- Вы можете объяснить, где серверное время, а где клиентское.
- Вы не доверяете `clientTime` без clamp.
- Локальный эффект не меняет авторитетный результат.
- Есть список механик, которым prediction реально нужен.

---

## Минимальная диагностика

| Симптом | Что проверить |
|---------|---------------|
| Игрок видит резкие откаты | Prediction есть, reconciliation грубый или нет smoothing. |
| Выстрел "попал на клиенте", но не на server | Нужна server-side lag compensation или другой combat design. |
| Latency test ничего не меняет | Тестируете localhost без latency/loss simulation. |
| Код стал неподдерживаемым | Вы внедрили prediction до базового stable movement. |

---

## Частые ошибки

- Копировать сложный prediction из статьи без понимания.
- Доверять timestamp клиента.
- Делать rewind для всего мира вместо конкретных hitboxes.
- Исправлять задержку полным client authority без проверок.

---

## Лайфхаки

- Для первого релиза лучше понятный server authority, чем полусломанный prediction.
- Visual feedback можно показывать сразу, авторитетный результат позже.
- В документации Mirror и примерах проверяйте код под тот же tag, что установлен в проекте.
- Все решения по movement заносите в `MOVEMENT_NET.md`.

---

## Профессиональный минимум

- У проекта есть latency budget: что допустимо для жанра.
- Prediction не включается без reconciliation.
- Server validation остаётся даже при client-side feel.
- Все решения проверяются на плохой сети, а не на идеальном localhost.
