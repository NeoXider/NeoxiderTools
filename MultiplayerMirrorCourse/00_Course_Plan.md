## 🧱 Блок A — Фундамент (ур. 1–2)

- Транспорты (KCP, Telepathy, SimpleWeb, Multiplex), топологии Host / Client / Dedicated.
- `NetworkManager`, кастомный менеджер, флаг `listen`, жизненный цикл.
- `NetworkIdentity`, `netId`, спавн, authority.

**Ссылки:** [00a](00a_Prerequisites_and_Networking_Basics.md) → [01](01_Mirror_Architecture.md) → [02](02_NetworkIdentity_and_Spawning.md).

**✅ Критерий готовности:** Два окна (ParrelSync или два билда), связь host + client, успешный спавн игрока, лог `connectionId`, выбранный транспорт задокументирован.

---

## 🔄 Блок B — Состояние и события (ур. 3–6)

- `NetworkTransform` (Reliable / Unreliable / Hybrid v96), интерполяция, движение.
- `SyncVar`, hooks, направление сервер → клиент.
- `SyncList`, `SyncDictionary`, `SyncHashSet`, сериализаторы.
- `[Command]`, `[ClientRpc]`, `[TargetRpc]`, rate limiting.

**Ссылки:** [03](03_Movement_and_Physics.md) → [04](04_State_Synchronization_SyncVars.md) → [05](05_Complex_Data_Structures.md) → [06](06_RPC_and_Commands.md).

**✅ Критерий готовности:** Движущаяся платформа на сервере; синхронизация HP с hook; инвентарь на `SyncDictionary`; покупка предмета через `Command` с серверной проверкой.

---

## 🌍 Блок C — Масштаб и сессии (ур. 7–8)

- Interest Management (Hex, Scene Distance v96), `SetHostVisibility`.
- `NetworkRoomManager`, лобби, смена сцен.

**Ссылки:** [07](07_Interest_Management.md) → [08](08_Matchmaking_and_Rooms.md).

**✅ Критерий готовности:** Замер трафика до/после IM на сцене с ботами; рабочее лобби с кнопкой Ready и безопасным переходом в игру.

---

## ⚖️ Блок D — Честность и ощущения (ур. 9–10)

- Серверная валидация, античит-мышление.
- Prediction, lag compensation.

**Ссылки:** [09](09_Server_Authority_AntiCheat.md) → [10](10_Lag_Compensation.md).

**✅ Критерий готовности:** Серверный raycast / урон без доверия клиенту; заметки по симуляции 150 ms; осознанный выбор уровня hitreg.

---

## 🖥️ Блок E — Архитектура клиента (ур. 11–12)

- ScriptableObjects, сетевые каталоги ID.
- Сетевой UI, disconnect, отделение логики от `NetworkBehaviour`.

**Ссылки:** [11](11_ScriptableObjects_and_Network.md) → [12](12_Network_UI.md).

**✅ Критерий готовности:** Передача только id по сети + локальный каталог; всплывающая панель disconnect.

## 🚀 Блок F — Продакшн (ур. 13–15)

- Оптимизация, профилирование трафика, сжатие.
- Dedicated, headless серверы, `UNITY_SERVER`.
- Дебаг, симуляция проблем сети, чек-лист релиза.

**Ссылки:** [13](13_Optimization_and_Profiling.md) → [14](14_Dedicated_Server.md) → [15](15_Debugging_and_Release.md).

**✅ Критерий готовности:** Headless билд + удаленный коннект; чек-лист релиза пройден с искусственной симуляцией задержки/потерь.

## 🔗 Точки интеграции между уроками (Шпаргалка)

| Если вы… | То следующий шаг |

## ❓ FAQ по плану

**Вопрос:** Можно ли пропускать уроки?  
**Ответ:** Только осознанно! Например, Steam (18) не нужен, если вы не планируете релиз в Steam, но основы NAT (16) критически полезны всем.

**Вопрос:** С чего начать, если дедлайн для прототипа — две недели?  
**Ответ:** Маршрут выживания: `00a` → `1` → `2` → `6` → `9` → `15` → `22`. Это минимальный скелет для билда «играет по сети и не стыдно показать».

**Вопрос:** А где финальный проект (капстоун)?  
**Ответ:** Это комбинация домашних заданий уроков 8, 15, 22 и 30: связка лобби + чек-лист релиза + smoke + стратегия сетевого стека для вашей реальной игры.

## ⚠️ Риски обучения (и как их снять)

| Риск | Мера |
|------|------|
| Теория без реального коннекта (только 1 клиент) | Использовать ParrelSync с первой же недели. |
| Преждевременная оптимизация | Не приступать к блоку F без профилирования (замеров). |
| Застревание на выборе транспорта | Пройти урок 1 и выбрать **один** путь (например, KCP) до первого playable билда. |
