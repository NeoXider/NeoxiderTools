# No-Code в NeoxiderTools: аудит, границы и варианты решений

| Поле | Значение |
|------|----------|
| Дата | 2026-05-03 |
| Контекст | Больше связок **данные ↔ UI ↔ физика** без C#; при этом **визуальное программирование** (ноды, графы, «сборка логики») **не** является целевым No-Code — это остаётся формой программирования. |
| Версия пакета | см. [`package.json`](../package.json) |

---

## 1. Что мы называем No-Code здесь

**Цель:** типичный пользователь пакета настраивает поведение через **Inspector**, **готовые компоненты**, **данные** (ScriptableObject, ключи сохранения, пресеты) и **ограниченный каталог действий** — без написания скриптов и без сборки программы из блоков.

**Не входит в определение (явный non-goal):**

| Подход | Почему не цель |
|--------|----------------|
| Нодовые редакторы, Blueprints, «цепочки блоков» с ветвлениями | Это **визуальное программирование**: те же абстракции, что и в коде (ветвление, переменные, типы), только на экране. |
| Произвольный вызов любого метода любого типа по строке из Inspector | Риск, поддержка, по сути **скриптование через строки**. |
| Полная универсальность «любой объект ↔ любое поле ↔ любое действие» без ограничений | Невозможно без рефлексии уровня языка или графа логики — противоречит разделу выше. |

**Что допустимо вместо «программирования»:**

- **Конфигурация** — заполнение формы, выбор из списка **проверенных** пресетов.
- **Данные как контент** — таблицы, ScriptableObject, ключи сохранения; изменение числа или строки ≠ написание логики.
- **Жёсткий каталог сценариев** — одно действие выбирается из выпадающего списка («нанести урон», «показать значение», «воспроизвести звук»), без составления новой логики из примитивов.
- **Связь по ссылкам и ролям** — «этот TMP показывает то поле того объекта», указанное через ObjectField и имя поля из выпадающего списка **известных** источников.

## 2. Текущее состояние (кратко)

| Область | Есть в пакете | Разрыв для «чистого» No-Code |
|---------|----------------|------------------------------|
| Условия | **NeoCondition** — поля, AND/OR, события | Нет симметричного универсального «событие → ограниченное действие над объектом» без мостовых скриптов |
| Деньги | **Money**, **TextMoney**, `_persistMoney`, сброс/перезагрузка из Save ([`Shop/Money.md`](./Shop/Money.md)) | Для **других** источников чисел всё ещё часто нужны отдельные «вью»-компоненты по образцу TextMoney |
| Текст | **SetText**, форматирование; привязка к полю/`ReactivePropertyFloat` — **Neo.NoCode** (`NoCodeBindText`) | Для **других** сцен без биндера всё ещё могут понадобиться точечные «вью»-компоненты |
| RPG UI | **RpgHpBarUI** и т.д. | Привязка к конкретным типам в коде; не масштабируется на произвольный компонент из Inspector |
| Физика | **PhysicsEvents2D/3D**, UnityEvent с коллайдером | Типичный запрос «урон по триггеру на Other» часто требует промежуточного Behaviour |
| Рефлексия | **ReflectionCache** (Neo.Condition); **Neo.NoCode** использует тот же кеш; резолв **`GameObject`** (ссылка / **Find** по имени / хост) — общий **`BindingSourceGameObjectResolver`** с NeoCondition | Не выделен единый слой **безопасных** действий для контактов и произвольных сценариев |

## 3. Варианты решений (включая новые подходы)

Ниже — **не** нодовые редакторы, а подходы уровня **данных и ограниченных форм**.

### 3.1 Универсальный биндинг значения → UI (данные + форма)

**Суть:** один компонент «Источник (объект + поле из белого списка или Reactive-адаптер) → приёмник (TMP / Slider / Image fill)». Настройка выглядит как **анкета**, а не как граф.

- Переиспользовать форматирование **SetText**.
- Рефлексия только при старте и при смене цели; кеш как в NeoCondition.
- **Отличие от визуального программирования:** нет ветвлений и составных программ — только «откуда взять число → куда вывести».

### 3.2 Пресеты физического контакта (каталог действий)

**Суть:** на объекте с триггером не «написать логику», а выбрать пресет: *«ApplyDamageToOther»*, параметры: урон, cooldown, фильтр по слою (уже есть в PhysicsEvents).

- Реализация ищет на **Other** компонент из **фиксированного списка** интерфейсов/типов (`IDamageable`, …), вызывает **один** согласованный метод или поднимает типизированный UnityEvent.
- **Не** универсальный «вызови любой метод по имени».

### 3.3 Правила как таблица (контент, не код)

**Суть:** для части игр достаточно **строк таблицы**: условие (ключ сохранения / тег / слой) → действие из закрытого перечня. Автор заполняет таблицу в Inspector или импортирует CSV; движок интерпретирует только известные колонки.

- Похоже на балансировочные таблицы, а не на скрипт.
- Ограничение: сложные ветвления таблицей неудобны — это осознанный компромисс.

### 3.4 Роли и слоты отображения (семантика без типов C#)

**Суть:** дизайнер помечает объекты ролями уровня игры: «слот отображения валюты», «слот полоски здоровья», «источник — компонент на этом же объекте / на цели». Связка выбирается из **enum ролей**, а не из произвольных типов.

- Снижает когнитивную нагрузку по сравнению с «Component type + field name».
- Реализация в коде сопоставляет роль конкретным типам один раз централизованно.

### 3.5 Мастер сцены / рецепты (генерация связок)

**Суть:** пункт меню «Добавить типичную связку: триггер урона / отображение Money» создаёт **готовую иерархию префабов** с уже назначенными ссылками. Пользователь **не видит** «программу», только заполненный Inspector.

- Это **авторинг окружения**, а не язык логики.

### 3.6 Единый слой «NeoInteraction» как **список шагов из каталога**

**Суть:** если нужна цепочка, это не граф, а **упорядоченный список** элементов вида: *[Включить объект, Воспроизвести клип, Установить поле X]* — каждый элемент из **фиксированного** набора типов шагов.

- Отличается от нодов: нет произвольных проводов между произвольными портами; только выбор из шаблонов и порядок в списке.

### 3.7 Согласование с NeoCondition

- Фильтрация контакта остаётся в **PhysicsEvents**; новые компоненты **подписываются** на уже существующие события.
- NeoCondition — «если условие»; действия после условия — из того же **закрытого каталога**, что и в п. 3.6.

## 4. Риски и принципы

1. Не выполнять тяжёлую рефлексию в **Update** — только при старте и смене цели.
2. **Безопасность Inspector:** типизированные UnityEvent и белые списки типов; строковые имена методов — только для advanced и с валидацией в Editor.
3. Не дублировать фильтры слоя/тега — композиция с **PhysicsEvents**.
4. Закладывать расширяемость под authority, если появится сеть.

## 6. Метрики успеха

- Без C# настроить: **вывод произвольного поддерживаемого числа на TMP** и **типовой урон по триггеру** к объекту с известным компонентом.
- Меньше однострочных мостовых скриптов в демо; рост Edit Mode тестов на биндинг и безопасность вызовов.
