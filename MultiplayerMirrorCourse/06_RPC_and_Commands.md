# Урок 6: Command, ClientRpc и TargetRpc

## Цели

Различать направления вызовов, безопасно принимать действия игрока на сервере, ограничивать частоту команд, знать где брать **`connectionId`** (на `NetworkConnectionToClient` в v96).

---

## Проблема и контекст

| Ошибка | Последствия |
|--------|-------------|
| Игровая логика только в `ClientRpc` | Клиент решает исход — чит |
| Нет кулдауна на `Command` | Спам RPC, DoS своего сервера |
| Путаница с `connectionId` | Неверный матчмейкинг / кик не того игрока |

---

## Теория

- **`Cmd`** — клиент → сервер (валидация обязательна).
- **`Rpc`** — сервер → все клиенты (эффекты, не состояние боя).
- **`TargetRpc`** — сервер → один `NetworkConnection`.

Префиксы имён методов обязательны (`CmdShoot`, `RpcFlash`).

**Authority у `Command` (Mirror):** по умолчанию `[Command]` эквивалентен `[Command(requiresAuthority = true)]`. Такую команду клиент может **инициировать только** с объекта, для которого выполняется `isLocalPlayer` или `isOwned` — иначе Mirror **не отправит** пакет на сервер и выдаст предупреждение вида *called without authority*. Это типичная ловушка UI на сценовом `NetworkBehaviour` без выданного authority: кнопка вызывает `Command`, а цепочка обрывается.

**Исключение:** `[Command(requiresAuthority = false)]` разрешает вызов **с любого** клиента, у которого активен `NetworkClient` и клиент **ready** — сервер всё равно исполняет тело метода, но **любой** подключённый клиент теоретически может слать эту команду. Используйте только осознанно: чат/лобби-контроллеры, прототипы, редкие сценовые триггеры. Всегда валидируйте на сервере (кто нажал, фаза матча, кулдаун), иначе получите уязвимость и спам. В продакшене чаще надёжнее повесить `Command` на **префаб игрока** с обычным `requiresAuthority = true` и передавать данные параметрами.

## Практика

```csharp
[Command]
void CmdJump()
{
    // Сервер: проверьте кулдаун и isGrounded по своей логике
    velocityY = jumpSpeed;
}

[ClientRpc]
void RpcPlayVfx() { /* particles */ }
```

Добавьте серверный таймер / счётчик на соединение против спама прыжков.

---

## Типичные ошибки

- Вызывать `Command` не с владельца объекта с authority.
- Доверять параметрам (`damage`, `price`) без проверки каталога на сервере.

---

## Советы и паттерны

- Тяжёлые проверки — один раз на сервере, результат — `SyncVar` или `TargetRpc` с итогом.
- Логируйте `conn.connectionId` при отладке лобби.

## Углублённо: `connectionId` в v96

На `NetworkConnectionToClient` доступен `connectionId` — используйте его в логах лобби и при выдаче токенов (урок 19). Не смешивайте с пользовательским `playerId` из БД: это разные пространства имён.

## Матрица направлений

| Атрибут | Откуда | Куда | Авторитет |
|---------|--------|------|------------|
| `[Command]` | клиент владельца (по умолчанию) | сервер | сервер решает; без authority см. `requiresAuthority = false` |
| `[ClientRpc]` | сервер | все клиенты | только эффекты |
| `[TargetRpc]` | сервер | одно соединение | доставка точечная |

## Валидация параметров

Любой `int itemId` проходит через каталог на сервере: существует ли предмет, доступен ли в этой зоне, хватает ли валюты, не превышен ли лимит в стаке.

## FAQ

**Вопрос:** Можно ли вызывать `ClientRpc` с клиента?  
**Ответ:** Нет в нормальной модели; только сервер инициирует широковещание.

**Вопрос:** Что за `requiresAuthority` у `Command`?  
**Ответ:** В Mirror это поле атрибута `[Command(requiresAuthority: bool)]`. Значение по умолчанию **`true`**: команду может отправить только владелец объекта (`isLocalPlayer` / `isOwned`). При **`false`** команду может вызвать любой клиент с активным соединением — удобно для сценовых менеджеров и прототипов, но на сервере обязательны проверки и лимиты. Не путать с устаревшими названиями из других API: в коде Mirror именно **`requiresAuthority`**.
