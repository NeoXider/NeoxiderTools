## Цели

Отделить «TCP подключился» от «игрока пустили в матч»; спроектировать выдачу короткоживущего токена с вашего backend.

---

## Проблема и контекст

| Без аутентификации | |
|--------------------|--|
| Любой клиент спавнит игрока | Боты, спам, утечки матчей |

---

## Теория

`NetworkAuthenticator` в Mirror: клиент присылает сообщение → сервер решает `Accept` / `Reject`. Токен обычно получают по **HTTPS** из вашего API (логин / платформа), затем передают в сообщении аутентификатору.

> **Важно:** секреты не вшивайте в клиентский билд; доверяйте только серверной проверке токена и сроку жизни.

---

## Практика

1. Наследник `NetworkAuthenticator` + регистрация в `NetworkManager`.
2. Контракт: структура `AuthRequestMessage` / `AuthResponseMessage` (имена по гайду вашей версии Mirror).

---

## Типичные ошибки

- Долгие блокирующие HTTP-запросы на главном потоке при логине — используйте async + таймаут.

---

## Советы и паттерны

Свяжите с rate limiting команд (урок 6): после auth всё равно лимитируйте `Command`.

---

## Домашнее задание

Опишите state machine: `Disconnected` → `Authenticating` → `InLobby` → `InMatch` и где обрывать соединение при провале. Добавьте **таймауты** на каждом переходе и текст ошибки для UI (урок 12).
