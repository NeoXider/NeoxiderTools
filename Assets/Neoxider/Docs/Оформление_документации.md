# Оформление документации

Правила оформления документации в NeoxiderTools: как описывать компоненты так, чтобы было удобно и при настройке в Inspector (No-Code), и при использовании из кода. Дополняет [DOCUMENTATION_GUIDELINES.md](../DOCUMENTATION_GUIDELINES.md) (структура папок, процесс написания).

---

## 1. Три слоя документации

Документация разделена на три слоя. Не дублируйте длинные тексты между ними: каждый слой — своя краткость.

| Слой | Где писать | Язык | Для кого | Что описывать |
|------|------------|------|----------|----------------|
| **Inspector** | Атрибуты `[Header]`, `[Tooltip]` в C# | **English** | No-Code, настройка в сцене | Кратко: что за поле/секция, единицы, поведение по умолчанию. |
| **XML в коде** | Комментарии `///` над классом, методом, свойством | **English** | Разработчики (IntelliSense) | Summary, при необходимости param/returns/remarks; для сложного — example/seealso. |
| **.md-файл** | `Docs/` (рус.) и `DocsEn/` (англ.), ссылка `[NeoDoc("...")]` → Docs | **Русский** / **English** | Все | Назначение, связи, поля, API, примеры No-Code и кода, «См. также». |

---

## 2. XML-документация в коде

- **Язык**: **English** (везде в коде: summary, param, returns, remarks, example, seealso).
- **Класс/компонент**: обязательно `/// <summary>` — зачем компонент, с чем связан; при необходимости `/// <remarks>` (когда использовать, ограничения).
- **Публичные методы**: `/// <summary>`, для каждого параметра — `/// <param name="name">`, при возврате значения — `/// <returns>`.
- **Публичные свойства**: `/// <summary>` или `/// <value>`.
- **Enum / константы**: по одному `/// <summary>` на элемент.
- **Опционально**: `/// <seealso cref="TypeOrMember"/>`, `/// <example>` — только для неочевидных случаев.

Пример:

```csharp
/// <summary>
///     Convenient driver for Animator parameters: trigger, bool, float, int. Use from code or wire to UnityEvent.
/// </summary>
public sealed class AnimatorParameterDriver : MonoBehaviour
{
    /// <summary>Fires the trigger by name.</summary>
    /// <param name="triggerName">Parameter name in the Animator.</param>
    public void SetTrigger(string triggerName) { ... }
}
```

## 3. Атрибуты Inspector (Tooltip, Header)

- **Язык**: **English** (все тексты в `[Header]` и `[Tooltip]`).
- У **каждого сериализуемого поля**, видимого в Inspector, должен быть `[Tooltip("...")]`: что это, единицы/диапазон, что будет, если оставить пустым.
- Группировка: `[Header("Section name")]` для блоков (например, "Settings", "Events", "Save").
- Текст Tooltip — одно короткое предложение.

Пример:

```csharp
[Header("Parameter names (for methods without name argument)")]
[SerializeField]
[Tooltip("Trigger parameter name for SetTrigger() when called with no argument.")]
private string triggerParameterName;
```

## 4. Шаблон .md-страницы компонента

Используйте эту структуру для каждого компонента с собственным .md-файлом.

### 4.1. Обязательные блоки

1. **Заголовок** — имя компонента (H1).
2. **Назначение** — 1–3 предложения: зачем компонент, какую задачу решает.
3. **Поля (Inspector)** — таблица: имя поля (как в Inspector), тип/единицы, краткое описание. Группируйте по секциям `[Header]`.
4. **API** — публичные методы и свойства, удобные для вызова из кода и из UnityEvent; возвращаемые значения.
5. **Unity Events** — когда вызывается каждое событие, какие параметры передаёт (важно для No-Code).
6. **Примеры** — минимум два: настройка в сцене/UnityEvent (No-Code) и короткий фрагмент C# (код).
7. **См. также** — ссылки на связанные документы и обратная ссылка на раздел (например, «← [Tools/Components](README.md)»).

### 4.2. Опциональные блоки

- **Связи и зависимости** — с чем работает (Animator, SaveProvider и т.д.), требуемые компоненты, ссылки на связанные .md.
- **Установка/подключение** — если нужны префаб, asmdef или особые шаги при первом запуске.

### 4.3. Краткая форма для простых компонентов

Для простых скриптов достаточно: **Назначение** → **Поля** (таблица) → **API** (таблица) → **Примеры** (No-Code + код) → **См. также**.

### 4.4. Пример таблицы полей

```markdown
## Поля

| Поле | Описание |
|------|----------|
| **Animator** | Целевой Animator. Если не задан — берётся с этого объекта. |
| **Trigger Parameter Name** | Имя триггера для метода SetTrigger() без аргумента. |
```

### 4.5. Пример таблицы API

```markdown
## API

| Метод | Описание |
|-------|----------|
| **SetTrigger()** | Вызвать триггер из поля Trigger Parameter Name. |
| **SetBool(string parameterName, bool value)** | Установить bool-параметр по имени. |
```

## 5. Язык и тон

- **В коде (XML, Inspector)**: только **English** — `/// summary`/param/returns, `[Tooltip]`, `[Header]`. Имена типов и методов в C# — английские.
- **В .md-файлах**: **русский** в `Docs/` (основная документация, ссылка из `[NeoDoc]`); **English** в `DocsEn/` (те же пути, что в `Docs/`, для англоязычной аудитории).
- **Тон**: кратко и по делу; не упускать важное для выбора «настроить в Inspector» или «вызвать из кода».

---

## 6. Связь кода и .md

- В классе компонента указывайте путь к **русскому** .md атрибутом: `[NeoDoc("Tools/Components/AnimatorParameterDriver.md")]`.
- Путь задаётся относительно папки `Assets/Neoxider/Docs/` (без ведущего слеша). В инспекторе Neoxider отображается блок документации с превью и кнопкой открытия этого .md.
- **Английские** страницы кладите в `Assets/Neoxider/DocsEn/` с тем же относительным путём (например, `DocsEn/Tools/Components/AnimatorParameterDriver.md`). Структура папок — как в `Docs/`. См. [DocsEn/README.md](../DocsEn/README.md).

---

## 7. Чеклист перед коммитом

- [ ] У всех видимых в Inspector полей есть `[Tooltip]`.
- [ ] Секции сгруппированы через `[Header]`.
- [ ] У класса есть `/// <summary>` (и при необходимости `/// <remarks>`).
- [ ] У публичных методов есть `/// <summary>`, у параметров — `/// <param>`, у возвращаемых значений — `/// <returns>`.
- [ ] Есть .md-файл с назначением, полями, API, событиями, примерами и «См. также».
- [ ] На классе указан `[NeoDoc("...")]` с путём к этому .md.

## 8. Готовность модуля (Definition of Done)

Модуль считается полностью задокументированным, когда выполнены **четыре вида**:

| Вид | Описание |
|-----|----------|
| **XML (EN)** | Все публичные типы/члены в модуле имеют XML-комментарии на английском. |
| **Inspector (EN)** | У всех сериализуемых полей есть `[Tooltip]` и при необходимости `[Header]` на английском. |
| **Docs (RU)** | В `Docs/` есть .md-файлы по [шаблону](#4-шаблон-md-страницы-компонента) на русском. |
| **DocsEn (EN)** | В `DocsEn/` — те же пути и структура, текст на английском (адаптированный перевод или 1:1). |

Контроль готовности по модулям ведите в рабочем чеклисте команды (issue/tracker). Этот файл удаляется, когда в проекте завершены все четыре вида документации для каждого модуля.
