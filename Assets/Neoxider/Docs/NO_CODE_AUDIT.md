# Аудит No-Code / Inspector-драйвен UX NeoxiderTools

| Поле | Значение |
|------|----------|
| Дата | 2026-05-03 |
| Контекст | Запрос на «полный ноукод»: меньше одноразовых скриптов, больше связок данных ↔ UI ↔ физика без кода |
| Версия пакета (ориентир) | см. `Assets/Neoxider/package.json` |

## 1. Цель аудита

Сформулировать **разрывы** между заявленным No-Code (NeoCondition, UnityEvent-компоненты, Inspector) и тем, что для типичных задач всё ещё нужны **мостовые скрипты** или **дублирующие «вью»** (отдельный компонент под каждый тип данных).

Документ предназначен для roadmap; реализация по приоритетам ниже.

---

## 2. Текущее состояние (кратко по кодовой базе)

| Область | Что уже есть | Ограничение для «чистого» No-Code |
|---------|----------------|-------------------------------------|
| Условия | **NeoCondition** — рефлексия полей/свойств, логика AND/OR, события | Это **проверка и события**, не универсальный «вызов действия по контакту» и не биндинг UI к любому источнику |
| Деньги / Shop | **Money** + **TextMoney** (наследник **SetText**), подписка на **ReactiveProperty** через код в компоненте | Есть готовый паттерн только для валюты; другие числовые/реактивные источники требуют **аналогичных отдельных** компонентов или ручной подписки |
| Текст / числа | **SetText**, форматирование чисел | Нет **универсального** «выбери объект → компонент → поле reactive/float → TMP_Text» без нового скрипта |
| RPG UI | **RpgHpBarUI**, **RpgLevelTextUI** и др. — привязка к конкретным **ReactiveProperty*** типам в коде | Хорошо для RPG-модуля; не масштабируется на любой произвольный компонент из Inspector без расширения API |
| Физика | **PhysicsEvents2D/3D** — фильтры слой/тег, **UnityEvent** с `Collider`/`Collision` | Событие передаёт коллайдер; чтобы «нанести урон по триггеру», нужно либо **отдельный скрипт** на колбэке, либо цепочка объектов в Inspector — без универсального «найди компонент и вызови метод/измени поле» |
| Рефлексия | В Condition есть кеширование и контур проверки значений | Инфраструктура **не переиспользована** как единый «No-Code действие» для контактов или для отображения |

## 3. Выявленные проблемы (сопоставление с запросом пользователя)

### 3.1 Деньги ↔ TMP и «вьюшки» не везде

**Симптом:** при изменении **Money** уже есть **TextMoney**, но для других систем (HP из произвольного компонента, счётчик очков в **SaveProvider**, квестовый прогресс, таймер и т.д.) нет единого Inspector-компонента «подписка на изменение → обновить TMP/Slider/Image».

**Последствие:** либо копируются паттерны **TextMoney**/**SetText**, либо пишутся маленькие скрипты «на объект».

**Направление улучшения (концепты):**

1. **Универсальный биндер значения → визуал** (рабочее имя: `NeoBind`, `ReactiveBind`, `InspectorValueBinder`):
   - источник: `UnityEngine.Object` + тип компонента + имя поля/свойства **или** явная ссылка на **ReactiveProperty**-совместимый объект (если доработать контракт отображения через интерфейс/адаптер);
   - приёмник: `TMP_Text`, опционально Slider/Image через делегаты форматирования;
   - переиспользовать форматирование из **SetText** (разделители, суффиксы).

2. **Слой совместимости с Reactive:** единый способ в Inspector указать «подписаться на `OnChanged`» без знания конкретного типа `ReactivePropertyFloat` vs `ReactivePropertyInt` (генерики не сериализуются как поля — нужны **обёртки** или **ScriptableObject**-дескриптор источника).

3. **Документация и меню:** один вход в меню создания «Bind value to text», демо-сцена с таблицей сценариев.

**Риски:** производительность при частом обновлении; нужно избегать рефлексии в Update и кешировать **MemberInfo** как в NeoCondition.

---

### 3.2 Триггер / коллизия → урон без отдельного скрипта

**Симптом:** **PhysicsEvents3D** отлично отдаёт коллайдер в **UnityEvent**, но типичный запрос «на триггер применить урон к **Health**/ **RpgCombatant** на объекте Other» требует промежуточного Behaviour.

**Направление улучшения:**

1. **NeoCollisionAction** / **InspectorPhysicalInteraction** (рабочее имя):
   - на том же объекте или на получателе из коллайдера: режим **«искать компонент на Other»** по имени типа или `[SerializeReference]` + выпадающий список известных интерфейсов (`IDamageable`, `IRpgCombatReceiver` и т.д.);
   - действие: **вызов метода** с параметром (float damage), **или** установка поля, **или** **UnityEvent** с уже разрешённым `Component`;

2. Параметры урона, интервал (cooldown на объект), фильтр по тегу уже частично есть в **PhysicsEvents** — не дублировать, а **компоновать** (один модуль фильтрации, второй — действие).

3. Согласование с **NeoCondition**: опционально «выполнять действие только если условие» (подмодуль или два компонента на одном GO).

**Риски:** безопасность вызова произвольных методов через Inspector — ограничить белым списком типов или только **UnityEvent** + фиксированный набор пресетов («ApplyDamage», «AddForce»).

---

### 3.3 Единый «No-Code действие» рядом с NeoCondition

Сейчас **NeoCondition** закрывает **если true/false → событие**. Нет симметричного первоклассного **«при событии X → выполни действие Y над объектом Z»** для физики, таймеров, клика по UI без отдельных скриптов.

**Идея:** модуль **NeoAction** / **NeoInteraction** (не путать с игровым Action):

- входы: источник события (PhysicsEvents уже шлёт UnityEvent — можно подключать), таймер, кнопка, NeoCondition OnTrue;
- действие: список шагов или один шаг: **SetProperty**, **CallMethod** (ограниченный), **Spawn**, **PlayAudio**, **EnableObject**;
- реализация рефлексии — общая библиотека с **ReflectionCache** из Condition.

## 4. Дополнительные направления (вне исходного примера, но полезны для «полного» No-Code)

| Направление | Зачем |
|-------------|--------|
| **Квест / Progression → TMP** | Отображение целей и чисел без кода, аналогично Money |
| **Timer / TimerObject → Slider + TMP** | Один компонент «оставшееся время» |
| **GM / G state → UI** | Частично есть через события; унифицировать «показать состояние как текст» |
| **Inventory** | Bind количества предмета к тексту по item id |
| **Локализация** | При бинде текста учитывать ключ локализации (отдельная задача) |

## 5. Принципы проектирования (чтобы не повторить антипаттерны)

1. **Не выполнять тяжёлую рефлексию в Update** — только при старте и при смене цели.
2. **Безопасность Inspector:** по возможности **UnityEvent** с типизированными аргументами вместо произвольной строки имени метода; если строка — только для advanced и с валидацией в Editor.
3. **Чёткое разделение:** фильтрация контакта (слои/теги) уже в **PhysicsEvents**; новый компонент не должен копировать фильтры, а **подписываться** на их события.
4. **Совместимость с мультиплеером** (если появится): действия урона должны иметь опцию «только authority» — закладывать как расширяемость.

## 6. Приоритетный roadmap (рекомендация)

| Приоритет | Эпик | Ожидаемый эффект |
|-----------|------|------------------|
| P0 | Универсальный **Value → TMP** (с форматированием SetText + поддержка Reactive через адаптеры) | Закрывает боль «каждый раз новый вью-скрипт» для чисел и простых полей |
| P1 | **Контакт → действие** с поиском компонента на Other и пресетами урона | Закрывает боль триггера без отдельного скрипта |
| P2 | Общая подсистема **NeoAction** + переиспользование кеша рефлексии с Condition | Единый стиль No-Code для разных источников событий |
| P3 | Демо-сцена и страницы в **Docs** («No-Code cookbook») | Снижение порога входа |

---

## 7. Метрики успеха

- Новый пользователь может без C#: **показать деньги / произвольное float-поле на TMP**, **нанести урон по триггеру** к объекту с известным компонентом.
- Количество «однострочных мостовых» скриптов в демо-сценах снижается; рост покрытия EditMode-тестами на биндинг и безопасность вызовов.
